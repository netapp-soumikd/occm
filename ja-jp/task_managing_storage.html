既存のストレージの管理

2020年9月1日寄稿者 このページのPDFをダウンロード

Cloud Managerを使用すると、ボリューム、アグリゲート、およびCIFSサーバーを管理できます。また、容量の問題を回避するためにボリュームを移動するように求められます。

既存のボリュームの管理

ストレージのニーズの変化に応じて、既存のボリュームを管理できます。ボリュームを表示、編集、複製、復元、および削除できます。

ステップ

  1. [Work Environments]ページで、ボリュームを管理するCloud VolumesONTAPの作業環境をダブルクリックします。

  2. ボリュームを管理します。

    仕事 アクション

    ボリュームに関する情報を表示する

    ボリュームを選択し、[情報]をクリックします。

    ボリュームの編集（読み取り/書き込みボリュームのみ）

    1. ボリュームを選択し、[編集]をクリックします。

    2. ボリュームのスナップショットポリシー、NFSプロトコルバージョン、NFSアクセス制御リスト、または共有アクセス許可を変更し、[更新]をクリックします。

    カスタムスナップショットポリシーが必要な場合は、SystemManagerを使用して作成できます。

    ボリュームのクローンを作成します

    1. ボリュームを選択し、[クローン]をクリックします。

    2. 必要に応じてクローン名を変更し、[クローン]をクリックします。

    このプロセスにより、FlexCloneボリュームが作成されます。 FlexCloneボリュームは、書き込み可能なポイントインタイムコピーであり、メタデータに少量のスペースを使用するためスペース効率が高く、データが変更または追加されたときにのみ追加のスペースを消費します。

    FlexCloneボリュームの詳細については、 『 ONTAP 9論理ストレージ管理ガイド』を参照してください。

    スナップショットコピーから新しいボリュームにデータを復元する

    1. ボリュームを選択し、[スナップショットコピーから復元]をクリックします。

    2. スナップショットコピーを選択し、新しいボリュームの名前を入力して、[復元]をクリックします。

    オンデマンドでスナップショットコピーを作成する

    1. ボリュームを選択し、[スナップショットコピーの作成]をクリックします。

    2. 必要に応じて名前を変更し、[作成]をクリックします。

    NFSマウントコマンドを取得する

    1. ボリュームを選択し、[マウントコマンド]をクリックします。

    2. [コピー]をクリックします。

    iSCSIボリュームのターゲットiQNを表示する

    1. ボリュームを選択し、[ TargetiQN ]をクリックします。

    2. [コピー]をクリックします。

    3. IQNを使用して、ホストからLUNに接続します。

    基になるディスクタイプを変更する

    1. ボリュームを選択し、[ディスクの種類と階層化ポリシーの変更]をクリックします。

    2. ディスクの種類を選択し、[変更]をクリックします。

    Cloud Managerは、選択したディスクタイプを使用する既存のアグリゲートにボリュームを移動するか、ボリュームの新しいアグリゲートを作成します。

    階層化ポリシーを変更する

    1. ボリュームを選択し、[ディスクの種類と階層化ポリシーの変更]をクリックします。

    2. [ポリシーの編集]をクリックします。

    3. 別のポリシーを選択して、[変更]をクリックします。

    Cloud Managerは、選択したディスクタイプを階層化して使用する既存のアグリゲートにボリュームを移動するか、ボリュームの新しいアグリゲートを作成します。

    ボリュームを削除する

    1. ボリュームを選択し、[削除]をクリックします。

    2. もう一度[削除]をクリックして確認します。

既存の集計の管理

ディスクを追加し、アグリゲートに関する情報を表示し、それらを削除することにより、アグリゲートを自分で管理します。

あなたが始める前に

アグリゲートを削除する場合は、最初にアグリゲート内のボリュームを削除しておく必要があります。

このタスクについて

アグリゲートのスペースが不足している場合は、OnCommand SystemManagerを使用してボリュームを別のアグリゲートに移動できます。

ステップ

  1. [Work Environments]ページで、アグリゲートを管理するCloud VolumesONTAPの作業環境をダブルクリックします。

  2. メニューアイコンをクリックし、[詳細] > [詳細割り当て]をクリックします。

  3. 集計を管理します。

    仕事 アクション

    集計に関する情報を表示する

    集計を選択し、[情報]をクリックします。

    特定のアグリゲートにボリュームを作成する

    アグリゲートを選択し、[ボリュームの作成]をクリックします。

    アグリゲートにディスクを追加する

    1. アグリゲートを選択し、[ AWSディスクの追加]または[ Azureディスクの追加]をクリックします。

    2. 追加するディスクの数を選択し、[追加]をクリックします。

      アグリゲート内のすべてのディスクは同じサイズである必要があります。

    集計を削除する

    1. ボリュームを含まないアグリゲートを選択し、[削除]をクリックします。

    2. もう一度[削除]をクリックして確認します。

CIFSサーバーの変更

DNSサーバーまたはActiveDirectoryドメインを変更する場合は、Cloud Volumes ONTAPのCIFSサーバーを変更して、クライアントにストレージを提供し続けることができるようにする必要があります。

ステップ

  1. 作業環境で、メニューアイコンをクリックし、[詳細設定] > [CIFSセットアップ]をクリックします。

  2. CIFSサーバーの設定を指定します。

    仕事 アクション

    DNSプライマリおよびセカンダリIPアドレス

    CIFSサーバーの名前解決を提供するDNSサーバーのIPアドレス。

    リストされたDNSサーバーには、CIFSサーバーが参加するドメインのActive Directory LDAPサーバーとドメインコントローラーを見つけるために必要なサービスロケーションレコード（SRV）が含まれている必要があります。

    参加するActiveDirectoryドメイン

    CIFSサーバーに参加させたいActiveDirectory（AD）ドメインのFQDN。

    ドメインへの参加を許可された資格情報

    ADドメイン内の指定された組織単位（OU）にコンピューターを追加するのに十分な特権を持つWindowsアカウントの名前とパスワード。

    CIFSサーバーのNetBIOS名

    ADドメインで一意のCIFSサーバー名。

    組織単位

    CIFSサーバーに関連付けるADドメイン内の組織単位。デフォルトはCN = Computersです。
    AWS Managed MicrosoftADをCloudVolumes ONTAPのADサーバーとして設定する場合は、このフィールドにOU = Computers、OU = corpと入力する必要があります。

    DNSドメイン

    Cloud Volumes ONTAPストレージ仮想マシン（SVM）のDNSドメイン。ほとんどの場合、ドメインはADドメインと同じです。

    NTPサーバー

    [ Active Directoryドメインを使用する]を選択して、Active DirectoryDNSを使用してNTPサーバーを構成します。別のアドレスを使用してNTPサーバーを構成する必要がある場合は、APIを使用する必要があります。詳細については、 Cloud ManagerAPI開発者ガイドを参照してください。

  3. [保存]をクリックします。

結果

Cloud Volumes ONTAPは、変更内容でCIFSサーバーを更新します。

ボリュームの移動

容量使用率、パフォーマンスの向上、およびサービスレベルアグリーメントを満たすためにボリュームを移動します。

システムマネージャでボリュームを移動するには、ボリュームとデスティネーションアグリゲートを選択し、ボリューム移動操作を開始し、オプションでボリューム移動ジョブを監視します。 System Managerを使用する場合、ボリューム移動操作は自動的に終了します。

ステップ

  1. System ManagerまたはCLIを使用して、ボリュームをアグリゲートに移動します。

    ほとんどの場合、SystemManagerを使用してボリュームを移動できます。

    手順については、 ONTAP 9 Volume MoveExpressガイドを参照してください。

CloudManagerがアクションが必要なメッセージを表示したときにボリュームを移動する

Cloud Managerは、容量の問題を回避するためにボリュームの移動が必要であるが、問題を修正するための推奨事項を提供できないことを示すActionRequiredメッセージを表示する場合があります。これが発生した場合は、問題を修正する方法を特定してから、1つ以上のボリュームを移動する必要があります。

ステップ

  1. 問題を修正する方法を特定します。

  2. 分析に基づいて、容量の問題を回避するためにボリュームを移動します。

    • ボリュームを別のシステムに移動します。

    • 同じシステム上の別のアグリゲートにボリュームを移動します。

容量の問題を修正する方法を特定する

Cloud Managerが容量の問題を回避するためにボリュームを移動するための推奨事項を提供できない場合は、移動する必要のあるボリュームと、それらを同じシステム上の別のアグリゲートに移動するか、別のシステムに移動するかを特定する必要があります。

ステップ

  1. 容量制限に達したアグリゲートを識別するために、「必要なアクション」メッセージの詳細情報を表示してください。

    たとえば、詳細情報には、次のような内容が含まれている必要があります。集計aggr1が容量制限に達しました。

  2. アグリゲートから移動する1つ以上のボリュームを特定します。

    1. 作業環境で、メニューアイコンをクリックし、[詳細] > [詳細割り当て]をクリックします。

    2. 集計を選択し、[情報]をクリックします。

    3. ボリュームのリストを展開します。

    4. 各ボリュームのサイズを確認し、1つ以上のボリュームを選択してアグリゲートから移動します。

      将来の追加の容量の問題を回避するために、アグリゲート内のスペースを解放するのに十分な大きさのボリュームを選択する必要があります。

  3. システムがディスク制限に達していない場合は、ボリュームを同じシステム上の既存のアグリゲートまたは新しいアグリゲートに移動する必要があります。

    詳細については、 容量の問題を回避するためにボリュームを別のアグリゲートに移動するを参照してください。

  4. システムがディスク制限に達した場合は、次のいずれかを実行します。

    1. 未使用のボリュームを削除します。

    2. ボリュームを再配置して、アグリゲートの空き領域にします。

      詳細については、 容量の問題を回避するためにボリュームを別のアグリゲートに移動するを参照してください。

    3. 2つ以上のボリュームをスペースのある別のシステムに移動します。

      詳細については、容量の問題を回避するための別のシステムへのボリュームの移動を参照してください。

容量の問題を回避するためにボリュームを別のシステムに移動する

1つ以上のボリュームを別のCloudVolumes ONTAPシステムに移動して、容量の問題を回避できます。システムがディスク制限に達した場合は、これを行う必要がある場合があります。

このタスクについて

このタスクの手順に従って、次のアクションが必要なメッセージを修正できます。

容量の問題を回避するには、ボリュームを移動する必要があります。ただし、システムがディスク制限に達したため、CloudManagerはこのアクションを実行できません。
ステップ

  1. 使用可能な容量があるCloudVolumes ONTAPシステムを特定するか、新しいシステムをデプロイします。

  2. ソース作業環境をターゲット作業環境にドラッグアンドドロップして、ボリュームの1回限りのデータレプリケーションを実行します。

    詳細については、システム間でのデータの複製を参照してください。

  3. [レプリケーションステータス]ページに移動し、SnapMirror関係を解除して、レプリケートされたボリュームをデータ保護ボリュームから読み取り/書き込みボリュームに変換します。

    詳細については、データレプリケーションのスケジュールと関係の管理を参照してください。

  4. データアクセス用にボリュームを構成します。

    データアクセス用の宛先ボリュームの構成については、 『 ONTAP 9 Volume Disaster Recovery Express Guide』を参照してください。

  5. 元のボリュームを削除します。

    詳細については、既存のボリュームの管理を参照してください。

容量の問題を回避するためにボリュームを別のアグリゲートに移動する

容量の問題を回避するために、1つ以上のボリュームを別のアグリゲートに移動できます。

このタスクについて

このタスクの手順に従って、次のアクションが必要なメッセージを修正できます。

容量の問題を回避するには、2つ以上のボリュームを移動する必要があります。ただし、CloudManagerはこのアクションを実行できません。
ステップ

  1. 既存のアグリゲートに、移動する必要のあるボリュームに使用可能な容量があるかどうかを確認します。

    1. 作業環境で、メニューアイコンをクリックし、[詳細] > [詳細割り当て]をクリックします。

    2. 各集計を選択し、[情報]をクリックして、使用可能な容量（集計容量から使用済み集計容量を引いたもの）を表示します。

  2. 必要に応じて、既存のアグリゲートにディスクを追加します。

    1. アグリゲートを選択し、[ディスクの追加]をクリックします。

    2. 追加するディスクの数を選択し、[追加]をクリックします。

  3. 使用可能な容量のあるアグリゲートがない場合は、新しいアグリゲートを作成します。

    詳細については、集計の作成を参照してください。

  4. System ManagerまたはCLIを使用して、ボリュームをアグリゲートに移動します。

  5. ほとんどの場合、SystemManagerを使用してボリュームを移動できます。

    手順については、 ONTAP 9 Volume MoveExpressガイドを参照してください。

ボリュームの移動が遅くなる可能性がある理由

Cloud Volumes ONTAPに次の条件のいずれかが当てはまる場合、ボリュームの移動に予想よりも時間がかかる場合があります。

  • ボリュームはクローンです。

  • ボリュームはクローンの親です。

  • ソースまたは宛先のアグリゲートには、単一のスループット最適化HDD（st1）ディスクがあります。

  • Cloud Volumes ONTAPシステムはAWSにあり、1つのアグリゲートはオブジェクトに古い命名スキームを使用します。両方のアグリゲートは同じ名前形式を使用する必要があります。

    9.4リリース以前のアグリゲートでデータ階層化が有効になっている場合は、古い命名スキームが使用されます。

  • 暗号化設定がソースアグリゲートとデスティネーションアグリゲートで一致していないか、キーの再生成が進行中です。

  • -tiering-policyオプションは、階層化ポリシーを変更するためにボリューム移動で指定されました。

  • -generate-destination-keyオプションは、ボリューム移動で指定されました。
