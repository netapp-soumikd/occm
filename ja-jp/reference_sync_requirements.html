ソースとターゲットのネットワーキング

  • ソースとターゲットには、データブローカーへのネットワーク接続が必要です。

    たとえば、NFSサーバーがデータセンターにあり、データブローカーがAWSにある場合、ネットワークからVPCへのネットワーク接続（VPNまたはダイレクトコネクト）が必要です。

  • NetAppは、ネットワークタイムプロトコル（NTP）サービスを使用するようにソース、ターゲット、およびデータブローカーを構成することをお勧めします。 3つのコンポーネント間の時間差は5分を超えてはなりません。

ソースとターゲットの要件

ソースとターゲットが次の要件を満たしていることを確認します。

AWSS3バケットの要件

AWSS3バケットが次の要件を満たしていることを確認してください。

AWSS3でサポートされているデータブローカーの場所

S3ストレージを含む同期関係には、AWSまたはオンプレミスにデプロイされたデータブローカーが必要です。いずれの場合も、Cloud Syncは、インストール中にデータブローカーをAWSアカウントに関連付けるように求めます。

  • AWSデータブローカーをデプロイする方法を学ぶ

  • Linuxホストにデータブローカーをインストールする方法を学ぶ

サポートされているAWSリージョン

中国とGovCloud（US）リージョンを除く、すべてのリージョンがサポートされています。

他のAWSアカウントのS3バケットに必要なアクセス許可

同期関係を設定するときに、データブローカーに関連付けられていないAWSアカウントに存在するS3バケットを指定できます。

このJSONファイルに含まれるアクセス許可は、データブローカーがアクセスできるようにそのS3バケットに適用する必要があります。これらの権限により、データブローカーはバケットとの間でデータをコピーし、バケット内のオブジェクトを一覧表示できます。

JSONファイルに含まれる権限について次の点に注意してください。

  1. <BucketName>は、データブローカーに関連付けられていないAWSアカウントに存在するバケットの名前です。

  2. <RoleARN>は、次のいずれかに置き換える必要があります。

    • データブローカーがLinuxホストに手動でインストールされた場合、 RoleARNは、データブローカーをデプロイするときにAWS認証情報を提供したAWSユーザーのARNである必要があります。

    • CloudFormationテンプレートを使用してデータブローカーがAWSにデプロイされた場合、 RoleARNはテンプレートによって作成されたIAMロールのARNである必要があります。

      ロールARNを見つけるには、EC2コンソールに移動し、データブローカーインスタンスを選択して、[Description]タブから[IAMロール]をクリックします。次に、ロールARNを含むIAMコンソールの[概要]ページが表示されます。

AzureBlobのストレージ要件

AzureBlobストレージが次の要件を満たしていることを確認してください。

AzureBlobでサポートされているデータブローカーの場所

同期関係にAzureBlobストレージが含まれている場合、データブローカーは任意の場所に存在できます。

サポートされているAzureリージョン

中国、米国政府、および米国国防総省の地域を除くすべての地域がサポートされています。

AzureBlobとNFS / SMBを含む関係に必要な接続文字列

Azure BlobコンテナーとNFSまたはSMBサーバーの間に同期関係を作成するときは、CloudSyncにストレージアカウント接続文字列を提供する必要があります。

2つのAzureBlobコンテナー間でデータを同期する場合は、接続文字列に共有アクセス署名（SAS）を含める必要があります。 BlobコンテナーとNFSまたはSMBサーバー間で同期するときにSASを使用するオプションもあります。

SASは、Blobサービスとすべてのリソースタイプ（サービス、コンテナ、およびオブジェクト）へのアクセスを許可する必要があります。 SASには、次の権限も含める必要があります。

  • ソースBlobコンテナーの場合：読み取りとリスト

  • ターゲットBlobコンテナーの場合：読み取り、書き込み、リスト、追加、および作成

AzureNetAppファイルの要件

Azure NetAppファイルとの間でデータを同期する場合は、PremiumまたはUltraサービスレベルを使用します。ディスクサービスレベルが標準の場合、障害やパフォーマンスの問題が発生する可能性があります。

適切なサービスレベルの決定についてサポートが必要な場合は、ソリューションアーキテクトに相談してください。ボリュームサイズとボリューム層によって、取得できるスループットが決まります。

Azure NetAppFilesのサービスレベルとスループットの詳細をご覧ください。

Google CloudStorageバケットの要件

Google CloudStorageバケットが次の要件を満たしていることを確認してください。

Google CloudStorageでサポートされているデータブローカーの場所

Google Cloud Storageを含む同期関係には、GCPまたはオンプレミスにデプロイされたデータブローカーが必要です。 Cloud Syncは、同期関係を作成するときに、データブローカーのインストールプロセスをガイドします。

  • GCPデータブローカーをデプロイする方法を学ぶ

  • Linuxホストにデータブローカーをインストールする方法を学ぶ

サポートされているGCPリージョン

すべての地域がサポートされています。

NFSサーバーの要件

  • NFSサーバーは、NetAppシステムまたは非NetAppシステムにすることができます。

  • ファイルサーバーは、データブローカーホストがエクスポートにアクセスできるようにする必要があります。

  • NFSバージョン3、4.0、4.1、および4.2がサポートされています。

    サーバーで目的のバージョンを有効にする必要があります。

  • ONTAPシステムからのNFSデータを同期する場合は、SVMのNFSエクスポートリストへのアクセスが有効になっていることを確認してください（vserver nfs modify -vserver svm_name -showmount enabled）。

    onTAP 9.2以降、showmountのデフォルト設定が有効になっています。

ONTAPS3のストレージ要件

ONTAP S3 Storageを含む同期関係を設定する場合は、以下を提供する必要があります。

  • ONTAPS3に接続されているLIFのIPアドレス

  • ONTAPが使用するように設定されているアクセスキーと秘密キー

SMBサーバーの要件

  • SMBサーバーは、NetAppシステムまたは非NetAppシステムにすることができます。

  • ファイルサーバーは、データブローカーホストがエクスポートにアクセスできるようにする必要があります。

  • SMBバージョン1.0、2.0、2.1、3.0、および3.11がサポートされています。

  • ソースフォルダとターゲットフォルダへの「フルコントロール」権限を持つ「Administrators」グループに付与します。

    この権限を付与しない場合、データブローカーはファイルまたはディレクトリでACLを取得するための十分な権限を持っていない可能性があります。これが発生した場合、次のエラーが表示されます：「getxattrエラー95」

隠しディレクトリと隠しファイルのSMB制限

SMBの制限は、SMBサーバー間でデータを同期するときに隠しディレクトリとファイルに影響します。ソースSMBサーバー上のディレクトリまたはファイルのいずれかがWindowsで非表示になっている場合、hidden属性はターゲットSMBサーバーにコピーされません。

大文字と小文字を区別しない制限によるSMB同期動作

SMBプロトコルでは大文字と小文字が区別されません。つまり、大文字と小文字は同じものとして扱われます。同期関係にSMBサーバーが含まれていて、データがターゲットに既に存在する場合、この動作により、ファイルの上書きやディレクトリコピーエラーが発生する可能性があります。

たとえば、ソースに「a」という名前のファイルがあり、ターゲットに「A」という名前のファイルがあるとします。 Cloud Syncが「a」という名前のファイルをターゲットにコピーすると、ファイル「A」はソースからのファイル「a」によって上書きされます。

ディレクトリの場合、ソースに「b」という名前のディレクトリがあり、ターゲットに「B」という名前のディレクトリがあるとしましょう。 Cloud Syncが「b」という名前のディレクトリをターゲットにコピーしようとすると、CloudSyncはディレクトリがすでに存在することを示すエラーを受け取ります。その結果、CloudSyncは常に「b」という名前のディレクトリのコピーに失敗します。

この制限を回避する最善の方法は、データを空のディレクトリに同期することを確認することです。

SnapMirror宛先のアクセス許可

同期関係のソースがSnapMirror宛先（読み取り専用）である場合、ソースからターゲットにデータを同期するには、「読み取り/リスト」権限で十分です。
